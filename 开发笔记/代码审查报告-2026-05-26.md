## 二、潜在问题

### 1. 设计缺陷

| 文件 | 问题描述 |
|------|----------|
| **blogger.gd** | 大量属性使用 `GDManager.get_blogger()` 代理访问，增加了耦合度且调试困难 |
| **blogger.gd** | `tmp_v = 23` 全局变量用于默认值，设计不合理 |
| **TaskManager.gd** | 任务状态存储在 `task_states` 数组中，每次需要遍历检查，O(n) 复杂度 |
| **SaveManager.gd** | 序列化/反序列化逻辑分散，添加新字段需要同时修改两个方法 |
| **GlobalDataManager.gd** | `_init()` 中调用 `_load_all_data()` 可能过早（节点树尚未建立） |
| **yun.gd** | 域名、主机、安全防护各自独立但相互依赖，状态管理复杂 |

### 2. 线程安全

| 文件 | 问题描述 |
|------|----------|
| **GlobalDataManager.gd** | 多处直接修改 `data_container` 中的数据，没有线程保护 |
| **SaveManager.gd** | `save_game()` 和 `load_game()` 如果在游戏运行时被调用可能造成数据竞争 |
| **ad_manager.gd** | `ad_money_0` 等变量在信号回调中修改，但没有同步机制 |

### 3. 内存泄漏风险

| 文件 | 问题描述 |
|------|----------|
| **blogger.gd L263** | `ViewsCalculator.new()` 每次 `update_blog_views()` 都会创建（如果为null），但从未释放 |
| **GlobalDataManager.gd L20** | `DataContainer.new()` 创建后未发现 `free()` 调用 |
| **yun.gd L23-26** | 初始化时创建多个子模块，没有发现清理逻辑 |
| **ad_manager.gd L189** | `ad_statistics` 数组无限增长，应该有清理机制 |

### 4. 资源管理

| 文件 | 问题描述 |
|------|----------|
| **milestones/config.gd** | 所有图标使用 `preload()` 在类加载时加载，大型项目会有启动延迟 |
| **blogger.gd** | `views_calculator` 是类成员变量，但如果游戏长时间运行，计算器的统计数据会无限增长 |

### 1. 性能优化

| 文件:行号 | 建议 |
|-----------|------|
| **blogger.gd:324-349** | `daily_activities()` 遍历任务列表使用 `Utils.check_name_exists()` 多次查询，应缓存结果 |
